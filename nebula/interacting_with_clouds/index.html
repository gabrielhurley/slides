Four Example Users

  1. End User
  2. Team/Project Administrator
  3. Cloud Administrator
  4. Executive

End User

  • Slightly technical
  • Trying to get their job done (their job is not the cloud)
  • Cloud means they don't have to wait six months for a server from procurement

Team/Project administrator

  • Moderately technical
  • Focused on enabling their team
  • Cloud means they have autonomous control over their allotted resources

Cloud administrator

  • Highly technical
  • Ensuring the stability and performance of the system
  • Cloud is their job

Executive

  • Not technical
  • Interested in discerning value and finding at-a-glance meaning

Your interface says a lot about you...


  • CLIs?

Command Line Interfaces

  • Often the most powerful/expressive
  • Great for scripting
  • Arcane (and poorly documented)
  • Intimidate, discourage and frustrate users
  • Often end up restricting their audiences

CLIS Don't Scale

  • Discoverability is very low
  • Lists only work for a few elements
  • At cloud-scale you mostly have machine-generated IDs
  • Interaction is limited by the nature of a command line

For Users

  • Copy-and-paste commands from Google searches that they mostly don't understand
  • CLIs get the job done, but they're frustrating
  • Why are they frustrating? Low discoverability and documentation that's overwhelming or over their head

For project administrators

  • CLIs provide the expressive power that's sometimes needed to accomplish deeper, less repetitive tasks
  • Still not the preferred tool due to complexity

For cloud administrators

  • Often the only way to get every last bit of control
  • Can be scripted into automated jobs
  • Requires investment of time to understand/master... but this is their job

For executives


Absolutely useless.

you really think they're gonna open a shell prompt? ;-)

What could be improved?

Imagine a highly-interactive CLI...

  • Type-ahead autocompletion of commands
  • Helpful doc strings with examples for every command
  • Layered documentation that can be broad or detailed
  • Autocompletion of names & ids
  • Meaningful colored output
  • "Shortcut" commands that trade power for ease-of-use

APIs (The RESTFUL Kind)


Underly everything else (CLIs, dashboards, etc.)

Good design is critical!

Your API is your limiting factor


Everything anyone builds will be based on your API!

  • And yet they're often an afterthought
  • They're built by developers for internal needs without thinking about consumers
  • They do "just enough" and no more

Important things nobody considers


  • Filtering
  • Ordering
  • Pagination
  • Fetching related data
  • Self-description (critical for federation)

For users


  • Need CLI tools or clients to interact with them
  • Easier to use than CLIs because you can use your language of choice
  • Just as badly documented as CLIs
  • Often missing capabilities that make integration infuriating

For project administrators


  • Same as for users, but perhaps less likely to need to integrate

For cloud administrators


  • Probably gonna just script things through the CLI or use a pre-existing tool
  • Sometimes build tools (and hopefully release them!)

For executives


Still useless.

Unless the API is awesome and people have already built tools for it...

Dashboards


Built on APIs!

Drastically increase discovery.

How do you see the world?


Dashbaords frequently take a "resource-oriented" view of the world.

  • Here are your servers, your volumes, etc.
  • Cover simple infrastructure management tasks

For users


  • Don't care about the specific instances, volumes; they want to get something done
  • Focus on workloads, applications, etc.

For project administrators


  • Need good overviews and utilization information
  • Easy management tools
  • Properly scoped interfaces

For cloud administrators


  • Lists are useless at "cloud scale"
  • Filtering is a start
  • Clustering is better
  • The API's the limit

For executives


  • Visualizations are key
  • Distillation of knowledge
  • Metrics, reporting and auditing

Mobile


CLIs... lol.

APIs... not likely.

Dashboards and Apps... yeah!

All about at-a-glance meaning


  • Progress/status of workloads
  • Monitoring health
  • Visualization
  • Identifying problems

For users


  • Nice-to-have if they exist
  • Primarily useful for progress indication, "done" notifications, retriggering jobs
  • Not likely to do advanced configuration of workloads on mobile

For project administrators


  • Quick management interface (add a user, increase a quota, etc.)
  • Quick utilization visualization
  • Problem reporting/resource allocation requesting ("going up the chain")

For cloud administrators


  • All about seeing utilization and health
  • Must be able to spot outliers (problems)
  • Respond to basic management requests

For executives


  • Pretty graphs
  • Roll-up reporting

Closing Thoughts & Common themes


  • How do you identify problems in a cloud?
  • What do you do when something goes wrong?
  • How do you learn?

Finding Problems


  • Monitoring systems
  • Proactive notification
  • Visualization
  • Outlier detection

Dealing with problems


  • Use a virtual console/remote desktop to check on it
  • SSH in and poke at it
  • Kill it and start over..
  • How much (if any) setup was done by hand?
  • Was data persisted or can it be rebuilt?
  • How long will it take to get back to running?

Learning (The Hard Part)


  • Need better discoverability
  • Need better documentation
  • Need clearer terminology
  • Need ready access to help
  • Teach a man to fish...
